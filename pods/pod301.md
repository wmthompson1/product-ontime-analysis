## 1. Baseline and Boundary Analysis (Testing the SQL-Only Claim)

Task #237 established a baseline baseline recommendation: keep all 7 MRP and inventory concepts restricted to pure-SQL snippets within `mrp_engine.py` and `solder_engine.py`. This blueprint re-examines that claim by parsing every concept into a strict classification matrix.

A graph topology provides excellent optimization for **routing and navigation** (traversing structural links across disparate operational entities). However, it cannot natively execute **row-level state predicates** (filtering on dynamic, mutable columns within a single record) or **time-phased bucket arithmetic** (cumulative rolling balances over time windows).

Below is the concrete boundary verification for each of the 7 concepts:

### 1. `AllocatedQuantity`

* **Classification:** (a) Routing/Navigation combined with (b) Row-Level State Predicate.
* **Boundary Test:** Determining *which* demand order has claimed *which* supply lot is a network routing problem that graph edges handle natively. However, calculating the actual magnitude of the allocation when a single work order is split or partially fulfilled requires evaluating dynamic scalar fields (`qty_ordered`, `qty_issued`). The graph can map the structural relationship, but the mathematical quantity remains a relational engine obligation.

### 2. `AvailableToPromise` (ATP)

* **Classification:** (c) Time-Phased Bucket Arithmetic.
* **Boundary Test:** ATP is defined mathematically at time $T_i$ as:

$$ATP(T_i) = \text{PlannedSupply}(T_i) - \sum \text{CommittedDemand}(T_j)$$



for all $T_j$ within the planning horizon until the next supply event. This calculation requires building sequential time buckets, ordering records chronologically, and computing running differences. Graph topology cannot carry this logic natively; an AQL traversal evaluating this would degenerate into an in-memory loop, whereas SQL window functions (`SUM() OVER (ORDER BY date)`) execute it via optimized relational vector paths.

### 3. `LeadTimeDemand`

* **Classification:** (b) Row-Level State Predicate combined with (c) Time-Phased Bucket Arithmetic.
* **Boundary Test:** This requires multiplying an average daily usage rate by a supplier's historical or static lead time value, then matching it against chronological consumption records. The calculation relies entirely on scalar multiplication and windowed aggregation over a specific historical duration—operations the graph cannot execute without falling back to embedded relational functions.

### 4. `SafetyStock`

* **Classification:** (b) Row-Level State Predicate.
* **Boundary Test:** Safety Stock is fundamentally a scalar parameter calculated either via a deterministic formula (e.g., max usage minus average usage) or set as a static configuration variable on an item master record. It represents a single threshold limit applied to a column value, offering zero routing or network navigation characteristics for graph topology to optimize.

### 5. `MinimumStockQuantity`

* **Classification:** (b) Row-Level State Predicate.
* **Boundary Test:** Like `SafetyStock`, this is a static or calculated threshold property mapped to an individual material or item row. It acts strictly as an evaluation constraint for inventory reorder alarms.

### 6. `MaximumStockQuantity`

* **Classification:** (b) Row-Level State Predicate.
* **Boundary Test:** A capacity or financial bounding constraint stored as a single literal field within the warehouse or item schema table. No multi-hop network evaluation exists.

### 7. `EconomicOrderQuantity` (EOQ)

* **Classification:** (b) Row-Level State Predicate.
* **Boundary Test:** EOQ utilizes the standard Wilson Formula:

$$EOQ = \sqrt{\frac{2DS}{H}}$$



where $D$ is annual demand quantity, $S$ is setup/order cost, and $H$ is holding cost per unit. This is entirely a row-level mathematical calculation executed over scalar configuration constants. It features zero graph-traversal patterns.

## 2. Hypothetical Edge Design for Graph-Topology Candidate Concepts

For the concepts where graph topology is structurally conceivable (`AllocatedQuantity` and `AvailableToPromise`), this section details how their candidate set-topology edges would be explicitly modeled under the project's strict architecture guidelines.

```text
 [Demand/SalesOrder] ───(mrp_allocates_to)───► [Supply/WorkOrder]

```

### 1. `AllocatedQuantity` Edge Design

* **Predicate / Edge-Type Name:** `mrp_allocates_to`
* **From Node Kind:** `concept__demand_order` (backed by `sales_order_line` or `dependent_demand` records)
* **To Node Kind:** `concept__supply_item` (backed by `purchase_order_line` or `work_order` records)
* **Cardinality:** Many-to-Many ($M:N$)
* **Edge Properties:** `{"allocated_qty": "NUMBER", "allocation_date": "STRING"}`
* **6-Slot Composite `_key` Layout:** Follows the project's deterministic string format:

$$\text{\_key} = \text{hash}(\text{from\_kind} \parallel \text{from\_id} \parallel \text{to\_kind} \parallel \text{to\_id} \parallel \text{predicate} \parallel \text{ordinal})$$


* **Two-Endpoint Prune Implication:** This edge explicitly spans two different underlying source tables (e.g., `sales_order_line` as the source endpoint and `work_order` as the target endpoint). Under our current graph pruning rules, if either table is truncated or modified during a localized development compilation pass, this edge would be flagged for deletion unless both endpoints are explicitly preserved within the query scope.

### 2. `AvailableToPromise` (Topology-Component) Edge Design

* **Predicate / Edge-Type Name:** `mrp_consumes_capacity`
* **From Node Kind:** `concept__supply_item`
* **To Node Kind:** `concept__work_center_capacity`
* **Cardinality:** Many-to-One ($M:1$)
* **Edge Properties:** `{"allocated_hours": "NUMBER", "planning_bucket": "STRING"}`
* **6-Slot Composite `_key` Layout:** Regular 6-slot hashing applying the reserved token `:CAPACITY` to ensure unique tracking blocks.
* **Two-Endpoint Prune Implication:** Spans the manufacturing schedule boundaries. If a plan pass runs for inventory models without loading the industrial work center capacity logs, the dependencies will fail validation checks.

## 3. End-to-End Canonical Graph Path Mechanics

If a pivot to graph topology were executed for these concepts, data would flow through our strict multi-layered promotion sequence. This path guarantees that no phantom nodes or untracked relationships creep into production.

```text
 [seed_elevations.py] ──► [export_graph_metadata.py] ──► [SCHEMA_VERSION Bump] ──► [load_canonical_to_arango.py]
                                                                                            │
                                                   ┌────────────────────────────────────────┘
                                                   ▼
                                        [ Dual Parity Gates ]
                                         ├── sql_graph_parity_check.py (Static File vs SQLite)
                                         └── sql_aql_parity_check.py  (SQLite vs Live Arango)

```

### 🔁 Step-by-Step Data Flow

1. **Edge Seeding:** The developer declares the edge relationships inside `replit_integrations/seed_elevations.py` using a structured Python manifest dictionary.
2. **Materialization Pass:** Running `replit_integrations/export_graph_metadata.py` scans the seed templates and inserts rows directly into the local SQLite schema tables: `sql_graph_nodes` and `sql_graph_edges`.
3. **The Snapshot Freeze:** The developer must manually increment the integer value `SCHEMA_VERSION` inside the project configuration. This action creates a permanent, frozen JSON backup file named `graph_metadata.v{N}.json`. This step introduces a strict management cost: **the snapshot cannot be modified post-merge**; any configuration drift requires generating a brand new version layer.
4. **Live Synchronization:** The deployment engine runs `replit_integrations/load_canonical_to_arango.py`. This reads the frozen snapshot file, connects to the ArangoDB cluster instance, and upserts the records into the live database collections.

### 🛡️ The Dual Parity Gates

To confirm that the virtual definition matches physical reality, the pipeline forces two separate validation passes:

* **Gate 1 (`sql_graph_parity_check.py`):** Ensures that the frozen `graph_metadata.json` file perfectly matches the raw node and edge configurations recorded inside the local SQLite database.
* **Gate 2 (`sql_aql_parity_check.py`):** Queries the local SQLite table layouts and runs comparison sets against the live ArangoDB cluster using AQL queries.

> ⚠️ **The Shared-Cloud Parity Race Caveat:** Because multiple CI runners connect to a shared development ArangoDB instance in the cloud, two parallel branch test runs can trigger a race condition—where Runner A mutates the collection states while Runner B is executing its assertions, causing intermittent validation dropouts.
> **The Real Acceptance:** **Gate 1 (`sql_graph_parity_check.py`) is the authoritative gate.** Because it evaluates purely deterministic local files against the local SQLite database context, it is free from multi-tenant cloud side-effects.

## 4. Ontop/SPARQL + SQLGlot Consequence Matrix

Integrating graph topology into the virtual knowledge layer alters our compilation pipeline, introducing distinct limits and capabilities across both the semantic engine and our query translation tools.

### 🏛️ Semantic Mappings & OWL Expressibility

To expose these hypothetical edge topologies through our virtualized SPARQL interface, the following extensions are required:

* **`.ttl` Additions:** Declare object properties `:allocatesTo` and `:consumesCapacity` as functional or inverse-functional constraints.
* **`.obda` Mappings:** Author explicit target-to-source SQL mapping pairs that transform our physical SQLite bridge tables into virtual RDF triples on demand:
```text
target    :demand/{order_id} :allocatesTo :supply/{item_id} .
source    SELECT order_id, item_id FROM sql_graph_edges WHERE predicate = 'mrp_allocates_to'

```


* **Expressibility Verification:** These basic edge relationships fall completely within the **OWL 2 QL profile**, allowing Ontop to rewrite graph interactions into standard relational inner joins without processing exceptions.

### 🛑 Ontop + SQLite Serialization Limits

* **The Single-Triple `OPTIONAL` Trap:** Ontop translates SPARQL `OPTIONAL` keywords into relational `LEFT JOIN` structures. SQLite handles basic left-outer expressions well, but nested or sequential `OPTIONAL` graph paths generate deep, un-indexed subqueries that degrade SQLite engine performance.
* **Aggregation Coalescing (`SUM`/`COUNT`):** Standard SPARQL aggregations drop `NULL` bindings silently. To prevent computational drift against our verified data schemas, our `.obda` SQL statements must explicitly inject `COALESCE(SUM(qty), 0)` overrides directly within the database access block.
* **The ATP Scalar-Split Gotcha:** Because ATP requires evaluating a running balance over calendar dates, attempting to write this in pure SPARQL forces a complex mixture of sub-selects and mathematical value tracking that Ontop cannot cleanly unfold into a single query execution pass, generating a runtime failure.

### 🤖 The SQLGlot "Lift" Requirement

When translating these virtual graph query blocks back into target physical models, our internal translation layer (`sql_lift.py`) must handle deep tree normalization:

* **Nested Left Join Unfolding:** SQLGlot must intercept deep, nested `LEFT JOIN` syntax blocks generated by Ontop's SPARQL compilation pass and automatically restructure them into optimized Common Table Expressions (CTEs) or flat inner-join sets.
* **CI Integration:** To safeguard this, an explicit validation step must be appended to `.github/workflows/ontop-interop-ci.yml`, running an automated test to prove that compiled SPARQL logic perfectly tracks our standard relational target outputs.

## 5. SQLGlot Dimension: Approved Snippets vs. M4 Computation Templates

Instead of building a graph topology or writing custom SQL strings for each database engine, we can store these 7 concepts using **Plain Approved Snippets** or promote them to **M4 Computation Template Metrics**.

### 1. Plain Approved Snippets (Current Architecture)

* **How it works:** Raw SQL snippets are authored in a dedicated directory and parsed by `solder_engine.py`.
* **Cross-Dialect Impact:** The snippets use standard ANSI SQL. However, if a snippet uses a specific dialect function (like SQLite's unique date manipulators), running that query against a production SQL Server or DuckDB instance will fail unless a developer manually creates and maintains a separate dialect file.

### 2. M4 Computation Template Metrics (Proposed Alternative)

* **How it works:** The calculation is declared once using a structured Python expression via an algebraic template definition.
* **Cross-Dialect Impact:** This approach provides complete dialect independence. The definition is parsed directly into an abstract syntax tree (AST). When deployed, SQLGlot automatically translates that AST into optimized, native SQL tailored for the target database:

```text
                 ┌──► [Transpile: DuckDB]  ──► Native Window Functions
 [ M4 Template ] ┼──► [Transpile: T-SQL]   ──► T-SQL Outer Apply Blocks
                 └──► [Transpile: SQLite]  ──► SQLite Date-Modifier Syntax

```

## 7. Architectural Verdict & Next Steps

### ⚖️ The Critical-Path Verdict

**Adopting a graph topology for any of these 7 concepts is officially rejected for the current development iteration.**

Attempting to implement a graph-edge approach for these metrics would immediately pull complex SPARQL unfolding work, `sql_lift.py` multi-nested join logic, and JVM-dependent compilation passes onto the critical path. This would delay our current delivery timeline while providing zero functional performance gains for calculations that are fundamentally numeric and time-phased.

### 🚀 Recommended Action Plan

1. **Proceed with Task #238:** Maintain the current development velocity by authoring the remaining MRP snippets as pure, relational SQL expressions.
2. **Isolate Calculations:** Keep static parameters (`SafetyStock`, `MinimumStockQuantity`, `MaximumStockQuantity`, `EconomicOrderQuantity`, `LeadTimeDemand`) as plain, optimized SQL snippets within `mrp_engine.py`.
3. **Plan the M4 Transition:** For the complex calculations (`AllocatedQuantity` and `AvailableToPromise`), schedule a future architecture task to move them from plain snippets into **M4 Computation Templates**. This approach ensures complete dialect portability across DuckDB and SQL Server without introducing graph database dependencies.
