The failures are coming from the masking tests for `staging.stg_vendor` and `staging.stg_part`. In the job logs, the assertions show `vendor_id` / `pref_vendor_id` are coming out as `None`, which means DuckDB isn’t producing a SHA-256 hex string for `SHA256(...)` in these models.

### Root cause
In DuckDB, `sha256()` returns a **BLOB** (bytes), not a hex-encoded string. Your tests expect a lowercase hex string like:

`6e890f636bd6fb774cec795520dd42e67b7f27691af024a5cd2a5afadf9b0c4c`

So the models need to **hex-encode** the digest (and ideally ensure the input is treated as text).

### Fix (code changes)

#### 1) Update `stg_vendor.sql`
File: `Utilities/SQLMesh/models/staging/stg_vendor.sql` at ref `773f754810421fb2cce0db00e7e72b81a8794132`  
Current line 24:
```sql
SHA256(id) AS vendor_id,
```

Replace with:
```sql
LOWER(HEX(SHA256(CAST(id AS VARCHAR)))) AS vendor_id,
```

#### 2) Update `stg_part.sql`
File: `Utilities/SQLMesh/models/staging/stg_part.sql` at ref `773f754810421fb2cce0db00e7e72b81a8794132`  
Current line 33:
```sql
SHA256(p.pref_vendor) AS pref_vendor_id,
```

Replace with:
```sql
LOWER(HEX(SHA256(CAST(p.pref_vendor AS VARCHAR)))) AS pref_vendor_id,
```

And update the join condition (lines 40–41) from:
```sql
ON SHA256(p.pref_vendor) = v.vendor_id
```

to:
```sql
ON LOWER(HEX(SHA256(CAST(p.pref_vendor AS VARCHAR)))) = v.vendor_id
```

### Why this resolves all 4 failing tests
- `test_vendor_id_is_hashed` / `test_vendor_two_rows_produce_distinct_hashes`: `vendor_id` will now be a deterministic hex string (matching the fixtures).
- `test_part_pref_vendor_is_hashed`: `pref_vendor_id` will match the expected hex string and also match `stg_vendor.vendor_id`.
- `test_part_with_unregistered_vendor_is_excluded`: the inner join will correctly drop orphan parts once the join uses the same hex-encoded hashing logic as `stg_vendor`.
