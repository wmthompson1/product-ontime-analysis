# **Part I — SQLMesh Fundamentals**  
## **Section 1 — What SQLMesh Is (Enterprise & Policy‑Aligned Definition)**

SQLMesh is a **declarative data transformation framework** that manages the full lifecycle of data models — including versioning, dependency analysis, backfills, testing, and environment promotion. It provides a consistent, reproducible way to define how data should be transformed, validated, and materialized across environments.

In your architecture, SQLMesh is not merely a modeling tool.  
It is the **firewall boundary** between:

- **Raw, PII‑bearing SQL Server tables** (inner zone)  
- **Hashed, masked, and aggregated tables** (outer zone)  
- **LLM‑driven agents** that must never see raw data  

This transforms SQLMesh from a convenience layer into a **policy‑enforcing compute perimeter**.

---

## **1.1 SQLMesh’s Core Purpose**

SQLMesh provides:

- A **declarative model layer** (SQL or Python)  
- A **plan engine** that determines what needs to be rebuilt  
- A **virtual environment system** that isolates changes  
- A **backfill engine** that ensures historical consistency  
- A **promotion mechanism** that moves validated results into production  

These capabilities allow SQLMesh to act as a **controlled, auditable transformation boundary** between sensitive data and downstream consumers.

---

## **1.2 SQLMesh as a Firewall in Your Architecture**

Your environment uses SQLMesh as a **software firewall** that enforces:

### **1.2.1 Controlled Ingress**  
SQLMesh is the **only component** allowed to read raw SQL Server tables.  
Models may reference upstream database tables directly — **no `.csv` seed files are required**.

This satisfies your policy question:

> **Yes, SQLMesh supports direct ingestion from database tables.**  
> Seed files are optional and not required for model inputs.

### **1.2.2 Controlled Egress**  
SQLMesh materializes only **hashed**, **masked**, and **aggregated** tables into the outer zone.  
These tables are the *only* objects accessible to:

- GitHubCopilotAgent  
- MCP skills  
- Any LLM‑driven automation  

This ensures no PII crosses the trust boundary.

---

## **1.3 SQLMesh’s Declarative Model System**

SQLMesh models define:

- **Inputs** (other models or external tables)  
- **Transformations** (SQL or Python)  
- **Materialization strategy** (table, view, incremental, etc.)  
- **Dependencies** (automatically inferred)  

In your environment, models fall into four layers:

1. **Raw ingest models** — direct SQL Server inputs  
2. **Hashed models** — PII removed, stable hashed keys introduced  
3. **Masked‑dimensionality models** — second‑pass reduction  
4. **Aggregated models** — safe for agent consumption  

This layered structure is central to your compliance posture.

---

## **1.4 SQLMesh Plans and Change Management**

SQLMesh’s plan engine evaluates:

- What changed  
- What depends on the change  
- What must be rebuilt  
- Whether historical backfills are required  

This is critical for your privacy model because:

- Any change to hashed or masked models triggers a **policy‑relevant review**  
- SQLMesh ensures **historical consistency** when dimensionality rules change  
- Plans provide an **audit trail** for compliance and governance  

Your agents will use SQLMesh plan inspection tools to generate:

- PR summaries  
- Privacy regression warnings  
- DAG impact reports  

But they will **never** run destructive plans in production.

---

## **1.5 Virtual Environments and Safe Promotion**

SQLMesh supports isolated environments such as:

- `dev`  
- `staging`  
- `prod`  

Changes are computed once, validated, and then **promoted without recomputation**.

This is ideal for your trust boundary:

- Raw data is touched only during initial computation  
- Promotion does not re‑read SQL Server  
- Agents only see the final hashed/masked/aggregated outputs  

---

## **1.6 Why SQLMesh Fits Your Policy Constraints**

Your two constraints are:

### **Constraint 1 — PII cannot leave SQL Server directly**  
SQLMesh satisfies this by:

- Reading raw tables internally  
- Producing hashed/masked/aggregated outputs  
- Exposing only those outputs to agents  

### **Constraint 2 — Masked dimensionality must be enforced**  
SQLMesh supports this by:

- Allowing a **second‑pass model layer**  
- Enforcing model contracts through code review  
- Recomputing historical data when dimensionality rules change  
- Providing DAG‑level visibility for privacy regression checks  

Together, these constraints turn SQLMesh into a **policy enforcement engine**, not just a transformation tool.
