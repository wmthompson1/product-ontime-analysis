The failures are coming from how the SHA-256 hash is being computed in DuckDB in your models.

In `staging.stg_vendor`, you currently do:

```sql
LOWER(HEX(SHA256(CAST(id AS VARCHAR)))) AS vendor_id
```

In DuckDB, `SHA256(<string>)` returns a **hex string already**. Wrapping it with `HEX(...)` converts that hex string into the hex encoding of its ASCII bytes, which is why the test output shows values starting with `36653839...` (that’s the ASCII-hex for `6e89...`), i.e., “double-hexing”.

The same issue exists in `staging.stg_part` for `pref_vendor_id`, and it also breaks your join guard (so the “unregistered vendor is excluded” test fails because the join doesn’t behave as intended).

## Fix

### 1) Update `Utilities/SQLMesh/models/staging/stg_vendor.sql`

Replace line 24:

```sql
LOWER(HEX(SHA256(CAST(id AS VARCHAR)))) AS vendor_id,
```

with:

```sql
LOWER(SHA256(CAST(id AS VARCHAR))) AS vendor_id,
```

### 2) Update `Utilities/SQLMesh/models/staging/stg_part.sql`

Replace line 33:

```sql
LOWER(HEX(SHA256(CAST(p.pref_vendor AS VARCHAR)))) AS pref_vendor_id,
```

with:

```sql
LOWER(SHA256(CAST(p.pref_vendor AS VARCHAR))) AS pref_vendor_id,
```

And replace the join condition (lines 40–41) from:

```sql
ON LOWER(HEX(SHA256(CAST(p.pref_vendor AS VARCHAR)))) = v.vendor_id
```

to:

```sql
ON LOWER(SHA256(CAST(p.pref_vendor AS VARCHAR))) = v.vendor_id
```

## Why this resolves all 4 failing tests

- `test_vendor_id_is_hashed` and `test_vendor_two_rows_produce_distinct_hashes` will now match the expected SHA-256 hex digest exactly (no extra `HEX()` encoding).
- `test_part_pref_vendor_is_hashed` will now produce the correct `pref_vendor_id`.
- `test_part_with_unregistered_vendor_is_excluded` will work because the join key calculation will match `staging.stg_vendor.vendor_id`, allowing valid rows through and excluding “orphan” vendors as intended.

After these edits, rerun `sqlmesh test --verbose` and the masking tests should pass.
