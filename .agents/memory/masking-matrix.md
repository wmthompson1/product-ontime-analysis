---
name: Masking matrix (column-masking DAG)
description: How the masking_matrix feature relates to schema, salt, and the CSV↔SQLite sync; what NOT to assume.
---

- The masking matrix (CSV `masking_matrix.csv` at the **repo root** ↔ SQLite `masking_matrix` table) is SEPARATE from `column_masking_policies` (the SME strategy tab). Both coexist by design; never collapse one into the other. The CSV was moved out of `certificate_for_receiving/` to the repo root so it sits beside its sibling `masking_type.csv` as the two SME-facing approval copies.
- `masking_type.csv` (root) ↔ SQLite `masking_type` table is a small controlled-vocabulary lookup (`masking_type,masking_mode,status`; CHECK status IN active/inactive) managed by `masking_type.py`, a faithful mirror of `masking_matrix.py` (same upsert-on-boot sync keyed on `masking_type`). The matrix's own `masking_type` column is the free-text label; this lookup is the vocabulary behind it — they are NOT foreign-keyed.
- Neither masking table is seeded in `schema_sqlite.sql` — first-run population comes ONLY from the committed root CSVs via the boot sync (`write_default_csv` recreates a CSV if it goes missing). **Why:** `init_sqlite_db` re-runs the schema script with every INSERT rewritten to `INSERT OR IGNORE` on EVERY boot, so a seed row an SME deleted via the UI would silently resurrect on the next restart (then get re-written to the CSV on the next save). **How to apply:** do NOT add seed INSERTs for these tables; deletes are made durable by the UI's full-replace writing the reduced CSV.
- The receiving certificate (`certificate_for_receiving/generate_certificate.py`) reads the **root** `masking_matrix.csv` and imports ONLY rows whose `status` is `active` (case-insensitive) — `static`/`complete` rows are intentionally excluded from the receiving cert. This active-only filter is the contract; do not loosen it.
- The "🧬 Masking Matrix" Gradio tab edits both grids and Save writes BOTH SQLite AND the root CSV (full-replace via `replace_matrix`/`replace_types`), so the SME approval copy and the live table never diverge. Salt status is shown but the salt value is NEVER printed.
- Its rows name columns (e.g. vendor.id, part.pref_vendor, user_def_fields) from the PRIVATE SQL Server schema (the synthetic twin's source), NOT this public SQLite twin — those tables/columns do not exist in `manufacturing.db`.
  **Why:** public/private repo duality + local SQLite columns are bare TEXT with no declared width, and SQL Server widths only appear inconsistently inside stored procs under `Data_Models/`. There is NO derivable local width source. **How to apply:** do NOT try to resolve the schema width by reading `manufacturing.db` or parsing DDL files — the width is carried in the matrix's own `field_length` column (see next bullet).
- The schema width lives in the matrix's `field_length INTEGER NOT NULL DEFAULT 0` column (0 = unbounded → full 64-char digest). `mask_row_value(row, value, salt)` sizes the hash to `row['field_length']`; `hash_sha256(value, length, salt)` is the lower primitive. `ensure_table()` self-heals older DBs via `ALTER TABLE ADD COLUMN` (every reader routes through it), so the executescript guard in app.py being a no-op on existing DBs is fine.
- `masking_rule` is a human-readable LABEL only — stored and copied verbatim into the receiving certificate but NEVER parsed. Masking is driven by `field_length` (width) + `masking_type`/`masking_mode`. It is now a uniform constant `hash_sha256(column_name,length)` on every row (was per-column like `hash_sha256(id,length)`). **How to apply:** changing the masking_rule text has zero functional effect; do not build a parser for it. If a CSV the SME hands you drops `field_length`, do NOT also drop it from the table — it's the only width source; preserve it and confirm intent.
- Masking salt lives only in the env/secret flow under `GEMIN_SALT` (overridable via `MASK_SALT_ENV`), confirmed by the user. Never stored in CSV/DB/code. `hash_sha256` raises rather than masking unsalted; output = uppercase SHA-256(value+salt) truncated to `field_length`; None/"" pass through.
- CSV→SQLite sync is upsert-only, keyed on `dag_no` (runs on every app boot via the lazy `get_db_engine`→`init_sqlite_db` path, wrapped so a missing/bad CSV never blocks boot).
  **Why:** simplicity + idempotent startup. **How to apply:** deleting a row from the CSV does NOT delete it from SQLite, and `replit_integrations/seed_masking_matrix.py --export` will resurrect a deleted row from the table back into the CSV. To truly remove a row, delete from SQLite first, then `--export`.
